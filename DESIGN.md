# 资产管理系统设计方向与迭代说明

本文面向后续接手产品、设计、前端、后端的人，帮助快速理解当前原型的设计意图。

## 1. 产品定位

本系统定位为企业 IT 资产全生命周期管理平台，不只是资产台账。

核心目标：

- 管理 IT 资产从采购、入库、领用、借用、归还、维修、调拨、盘点、报废到处置的完整生命周期。
- 覆盖设备、软件许可、耗材、配件、合同、供应商、盘点、故障维修等 ITAM 场景。
- 审批不自研流程引擎，而是对接成熟审批系统，例如飞书审批、泛微 OA、钉钉审批。
- 用户身份支持 OIDC 登录，登录后再绑定或创建本地用户，并由系统角色决定权限。

一句话：

> 本系统负责资产、单据、履历、权限和集成；外部系统负责认证和审批。

## 2. 当前原型范围

当前目录是纯前端静态原型，入口为 `index.html`，主要逻辑在 `app.js`，样式在 `styles.css`。

当前已经覆盖：

- 数据中控首页
- 资产管理工作台
- 资产分类、标签、状态、位置、风险筛选
- 资产详情抽屉
- 资产生命周期履历
- 管理员免审批直办
- 普通员工走外部审批
- 申请审批列表与审批轨迹
- 盘点任务
- 故障维修
- 耗材库存
- 合同供应商
- OIDC 身份源、Claim 映射、用户绑定

当前数据都是模拟数据，集中在 `app.js` 顶部的 `state` 对象中。

## 3. 设计原则

### 3.1 登录身份决定角色

不要在业务页面临时切换“管理员/员工视角”。

正确模型：

1. 用户通过账号密码或 OIDC 登录。
2. 系统识别或绑定本地用户。
3. 本地用户拥有一个或多个系统角色。
4. 角色决定菜单、数据范围、操作权限和是否可免审批直办。

当前原型默认登录用户为：

- 用户：体验用户1
- 账号：asset.admin
- 角色：资产管理员

资产页只展示当前登录角色，不提供切换按钮。

### 3.2 审批外接，不自研

系统不内置复杂审批流。

业务单据流转建议为：

1. 用户在资产系统创建业务单据。
2. 系统根据单据类型、角色和策略判断是否需要审批。
3. 需要审批时，创建飞书/泛微/钉钉审批实例。
4. 外部审批系统回调审批状态。
5. 审批通过后，资产系统执行资产动作并写入履历。

管理员直办场景：

- 资产管理员可对低风险、台账维护、盘点纠偏、紧急调拨等动作免审批直办。
- 直办必须写入操作日志和资产履历。
- 直办不是无痕操作。

### 3.3 OIDC 只负责认证

OIDC 登录后，外部身份信息不能直接等同于系统权限。

推荐绑定策略：

1. 优先使用 `sub` 绑定外部身份。
2. 其次使用 `email` 匹配本地用户。
3. 未匹配时自动新增普通员工，或进入待处理身份队列。
4. 本地角色由资产系统管理员分配。
5. 角色决定进入管理端还是员工端。

OIDC Claim 映射建议：

| OIDC Claim | 本地字段 | 用途 |
| --- | --- | --- |
| sub | externalSubject | 外部身份唯一 ID |
| email | email | 邮箱匹配和通知 |
| name | name | 用户姓名 |
| preferred_username | account | 登录账号 |
| department | department | 部门 |
| groups / roles | role | 可辅助映射角色，但不建议无审核直接授权高权限 |

### 3.4 资产管理要像 ITAM，不是普通表格

资产管理能力应包含：

- 资产分类：终端设备、基础设施、办公外设、网络设备、软件与许可等。
- 资产标签：二维码、RFID、许可池编号、业务标签。
- 生命周期：采购、入库、上架、领用、借用、归还、维修、调拨、下架、报废、处置。
- 责任模型：使用人、部门、保管人、成本中心、位置。
- 风险模型：过保、故障、配置待核验、即将续费、盘点异常。
- 字段完整度：用于发现台账质量问题。
- 审计留痕：所有关键变更都进入资产履历和操作日志。

## 4. 当前页面说明

### 数据中控

展示资产总数、待处理申请、资产原值、盘点差异，并提供常用操作入口。

后续应接入真实统计接口：

- 资产数量按状态、分类、部门统计
- 待处理单据
- 盘点异常
- 过保/即将过保
- 软件许可到期

### 资产管理

当前重点增强页面。

包含：

- 左侧资产分类
- 当前登录角色提示
- 资产 KPI
- 全文检索
- 状态、位置、风险筛选
- 标签筛选
- 资产表格
- 详情抽屉
- 管理员直办/员工申请动作

后端对接时建议拆分接口：

- `GET /assets`
- `GET /asset-categories`
- `GET /asset-tags`
- `GET /assets/:id`
- `POST /asset-actions`
- `POST /asset-actions/:id/execute-direct`
- `POST /asset-actions/:id/start-approval`

### 申请审批

展示本系统内的业务单据状态，不替代外部审批系统。

建议后端状态：

- 草稿
- 待提交审批
- 审批中
- 审批通过待执行
- 已完成
- 已驳回
- 已撤回
- 执行失败

### 组织权限

当前包含：

- OIDC 身份源
- 登录绑定策略
- 本地用户与外部身份绑定
- 角色权限矩阵
- Claim 字段映射
- 待处理身份

后续需要实现：

- 用户列表
- 角色列表
- 菜单权限
- 操作权限
- 数据范围
- OIDC Provider 配置
- 外部身份绑定/解绑

## 5. 建议后端领域模型

优先考虑这些核心实体：

- User
- ExternalIdentity
- Role
- Permission
- Department
- Asset
- AssetCategory
- AssetTag
- AssetLifecycleEvent
- AssetActionOrder
- ApprovalInstance
- ApprovalIntegration
- StocktakeTask
- StocktakeItem
- Supplier
- Contract
- Consumable
- OperationLog

其中 `AssetActionOrder` 是关键，它承载所有资产动作：

- 领用
- 借用
- 归还
- 调拨
- 维修
- 报废
- 处置
- 盘点纠偏
- 台账修正

审批只影响单据是否能执行，不直接替代业务单据。

## 6. 迭代建议

### 第一阶段：可演示 MVP

目标：让客户能完整体验资产管理闭环。

建议完成：

- 登录页
- OIDC 登录入口原型
- 资产台账
- 资产详情
- 资产新增/编辑
- 管理员直办
- 员工申请
- 外部审批模拟状态
- 操作日志

### 第二阶段：后端对接

目标：从静态原型变成真实可用系统。

建议完成：

- API 设计
- 用户和角色
- 资产 CRUD
- 资产动作单据
- 资产履历
- 基础权限控制

### 第三阶段：ITAM 深化

目标：形成差异化资产管理能力。

建议完成：

- 二维码/RFID 标签打印
- 批量导入导出
- 自定义字段
- 资产字段完整度
- 风险预警
- 保修/续费提醒
- 软件许可合规
- 盘点任务和移动端扫码

### 第四阶段：企业级能力

目标：支撑多部门、多系统、多租户或集团化管理。

建议完成：

- 多组织/多公司
- 数据权限策略
- 审计报表
- 开放 API
- Webhook
- 与 HR、财务、采购、CMDB、工单系统集成

## 7. 接手开发注意事项

- 不要把审批流写死在资产模块里，应抽象为审批适配层。
- 不要把 OIDC 外部角色直接等同于本地高权限角色，高权限应由本系统管理员确认。
- 管理员直办必须留痕。
- 普通员工侧应更简单，只展示本人资产和可申请动作。
- 资产履历应作为核心能力，不要只保留当前状态。
- 资产分类、标签、自定义字段要为后续扩展留接口。
- 原型目前是单文件 JS，正式开发建议拆成组件、路由、状态管理和 API 层。
